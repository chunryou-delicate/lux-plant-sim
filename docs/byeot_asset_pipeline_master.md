## 1-2. ★ 3D 친화 스타일 문구 (모든 프롬프트에 붙임)

```
[스타일] 파스텔 저폴리 3D 렌더, 민트·세이지·크림 파스텔 톤, 둥글둥글 말랑한 미니어처,
아이소메트릭 3/4 시점, 부드럽고 균일한 조명, 그림자 최소,
완전한 순백색 배경, 물체 하나만 정중앙, 깨끗한 실루엣, 동물의숲st 아기자기,
3D 에셋 변환용 (single object, clean silhouette, white background, even lighting, minimal shadow)
```

### ★★ 무늬·색변이는 "평평한 표면 색"으로 (실전에서 발견한 핵심 함정)
**문제**: 산세 무늬종 테스트에서 물결/줄무늬가 **도톰한 단차(요철)로 튀어나오게** 렌더됨.
무늬는 형태가 아니라 **표면 색(텍스처)** 이어야 하는데 AI가 색 경계를 입체 릴리프로 그림.
→ 이 원화를 3D로 뜨면 무늬가 울퉁불퉁한 지오메트리가 됨(잘못됨).

**해결**: 무늬·얼룩·줄무늬·변이색 들어가는 항목(무늬종·알보)엔 프롬프트에 **반드시** 추가:
```
무늬는 매끈하고 평평한 표면 위에 색으로만 표현(페인트처럼), 무늬 때문에 표면이 튀어나오거나 파이지 않음, 잎/표면은 하나의 매끄러운 형태.
(patterns are FLAT surface color only like painted decals, NOT raised or embossed geometry;
the surface stays smooth and continuous; variegation is texture, not shape)
```
> 좋은 예: 세로 줄무늬가 얇고 평평. 나쁜 예: 물결이 도톰한 단차.

## 1-3. ★ 자동 판정 루프 (사람 확인 없이 돈다)

**핵심: 클로드가 각 원화를 직접 눈으로 보고 합격/불합격 판정. 매번 안 묻는다.**

```
for 각 항목 in 배치:
    best = null
    for v in 1..4:                      # 최대 4회 리롤
        1. generate_image (프롬프트=개별+1-2번스타일, 1:1, count 1)
        2. 결과 이미지를 실제로 열어서 본다
        3. 1-4번 루브릭으로 채점 → PASS/FAIL + 점수(0~100)
        4. best 갱신 (최고점 보관)
        5. PASS면 → 즉시 종료, best 확정, 다음 항목
        6. FAIL이면 → 리롤 사유를 프롬프트에 반영해 v+1 재생성
    # 4회 안에 PASS 없으면:
    → 4장 중 best 채택(status=best_fallback), 넘어감. 멈추지 말 것.
    → best도 50점 미만이면 status=needs_review 로 남기고 진행
```

## 1-4. ★ 판정 루브릭 (사용자가 직접 정한 기준)

**필수 통과 (하나라도 어기면 FAIL → 리롤):**
- 순백 배경 + 물체 하나만 (배경 지저분/여러 물체/잘림 → FAIL)
- 파스텔 저폴리 스타일 일치 (사진st·리얼·다른 화풍 → FAIL)
- 단순명확한 실루엣 (뭉개짐/과한 디테일/애매한 형태 → FAIL)

**품질 평가 (점수, best 선정용):**
- 3D 변환 적합성 (그림자 최소·균일 조명·3/4각·입체 유추 쉬움)
- 현 게임(힐링 식물육성)에 맞는 톤 / 예쁘고 게임에 넣고 싶은 완성도
- 특징 구분 (산세=칼모양, 포토스=하트덩굴 등 종류 구별됨)
- 어색함 없음 (비율·왜곡·이상한 돌기 없음)
- ★ **무늬가 요철로 튀지 않았나** — 무늬가 도톰한 단차로 표현됐으면 감점, 심하면 FAIL+"무늬 평면화" 리롤. (저폴리 특성상 약한 면분할은 허용, 무늬 경계마다 확연히 튄 경우만 문제)
- 기존 에셋과 호환 (몬스테라 줄기·화분과 스케일·톤 맞음)

**★ 색은 봐준다:** 색만 어색한 건 FAIL 아님(나중에 텍스처로 조정 가능). PASS 시키고 note에 "색조정 필요"만. 단 스타일 자체를 깨는 색(네온·완전 사진st)은 FAIL.

**판정 기록**: 버전마다 `PASS/FAIL, 점수 N/100, 사유` (예: "FAIL 40 — 배경에 그림자·잎 2장").

## 1-5. 리롤 규칙 (자동 4회)
- 항목당 최대 4버전. 중간 PASS면 즉시 멈춤. 4회째면 best 채택하고 넘어감(멈추지 말 것).
- **한 배치에서 절반 이상 스타일 FAIL이면** → 개별 리롤 멈추고 "스타일 문구·모델 바꿔야 할 듯"이라고 사람에게 보고. (이건 예외적으로 멈춤)
- 리롤 시 프롬프트 교정 추가:
  - 배경 → "pure solid white background, nothing else"
  - 그림자 → "no cast shadow, flat even studio lighting"
  - 여러 물체 → "exactly one single object centered, nothing else in frame"
  - 무늬 요철 → "patterns are FLAT painted surface color only, NOT raised/embossed geometry; surface stays smooth and continuous"

## 1-6. 뽑을 배치 (순서)

> 프롬프트 원문은 아래 PART 4에 전부 있음. 각 프롬프트 = [개별 설명] + [1-2번 스타일 문구].

**배치 A — 아이템** (제일 쉽고 3D 잘 됨, 검증용): 물뿌리개 / 분무기 / 모종삽 / 전정가위 / 비료봉투 / 클립형 식물등 / 스탠드형 식물등

**배치 C — 식물** (★ 이번엔 3종 계열만): 몬스테라 / 몬스테라 알보 / 포토스 / 포토스 무늬종 / 산세베리아 / 산세베리아 무늬종
- 나머지 식물(다육·스파티·고무나무·상추·토마토 등)은 **보류** — 종별 상세검토 후 별도.
- ★ **잎 재활용**: 포토스 하트잎을 크게 키운 버전 = 몬스테라 중간잎(mid)으로 공용. 포토스 뽑을 때 "갈라짐 없는 큰 하트잎" 버전도 확보하면 몬스테라 중간잎 생성 아낌.

**배치 D — 화분**: 테라코타 / 회색 콘크리트 / 흰 도자기 / 검은 모종포트 / 유리 수경병 / 마크라메 행잉

**진행 순서 = A → C → D.** E(가구)·F(집)·G(GUI)·캐릭터는 이번 자동 대상 아님(A·C·D 끝나고 사용자와 결정). **캐릭터는 자동 대량 금지** — 일관성 때문에 한 장씩 참조로.

> **자동/수동 분기**: seedream이 1크레딧(유료)이라, 기본은 **배치마다 멈춰 사람 확인**.
> 단 사용자가 "그냥 자동으로 쭉 돌려"라고 하면 배치 끝까지 자동 진행 가능(크레딧 부담 적음).

## 1-7. 파일 정리 규칙

```
assets/                        (레포 내 실제 경로)
  ├── existing/                ← 사용자가 이미 뽑아둔 원화·이미지 (정리 대상, 아래 PART 2)
  ├── raw/                     ← 새로 뽑는 원화 저장
  │   ├── items/  plants/  pots/
  │   └── _rejected/           ← 탈락 원화 (삭제 말고 이동)
  ├── monstera/*.glb           ← 기존 몬스테라 GLB 13종 (done)
  └── manifest.json            ← 전체 대장
```
- 파일명: `{항목영문}_{v버전}.png` (소문자, 언더스코어). 예: `pothos_v1.png`
- **manifest.json 스키마** (항목마다):
```json
{
  "id": "wateringcan", "name_ko": "물뿌리개", "batch": "A_items",
  "prompt": "실제 사용한 전체 프롬프트", "model": "seedream_v4_5",
  "versions": [
    {"v":1, "job_id":"...", "url":"https://...", "local":"raw/items/wateringcan_v1.png",
     "status":"fail", "note":"FAIL 40 — 그림자 강함"},
    {"v":2, "job_id":"...", "url":"https://...", "local":"raw/items/wateringcan_v2.png",
     "status":"pass", "note":"PASS 82 — 깔끔"}
  ],
  "chosen_version": 2, "final_status": "approved", "credits_spent": 2.0
}
```
- status(버전): `pass` / `fail` / `best_fallback` / `needs_review`
- final_status(항목): `approved` / `needs_review` / `to_3d`(Meshy대기) / `done`(3D완료)

# ══════════ PART 3. 3D 변환 (Meshy) — 튜닝 실험 먼저 ══════════

> 대량 변환 전에, 설정값을 실제로 맞춰본다. 검증된 값으로 대량 돌림.

## 3-1. 시작 전 체크
- Meshy MCP 연결: `@meshy-ai/meshy-mcp-server` (Meshy 공식/오픈소스, 검증됨).
  - image_to_3d · remesh · GLB 다운로드 지원. `meshy_output/`에 로컬 자동 저장 → **로컬 실행 필수**.
  - 연결: `claude mcp add meshy -- npx -y @meshy-ai/meshy-mcp-server -e MESHY_API_KEY=<키>`
    (⚠️ API 키는 사용자가 터미널에서 직접 연결. 대화창엔 키 안 붙임.)
- GLB 뷰어: 윈도우 3D뷰어 or https://gltf-viewer.donmccurdy.com

## 3-2. 튜닝 대상 2개 (A/B — 무늬 처리 검증)
| # | 대상 | 목적 |
|---|---|---|
| T1 | 산세 무늬종 잎 (job `e42d9065...`) | 무늬가 **요철**로 그려진 원화 → 3D 뜨면 울퉁불퉁하나? remesh가 뭉개나? |
| T2 | 몬스테라 성숙 무늬종 잎 (kling 2749_1, existing 폴더) | 무늬가 **평평한** 좋은 원화 + 복잡한 갈라짐·구멍이 저폴리로 잘 뜨나? |
> T1(나쁜 원화)·T2(좋은 원화) 같은 설정으로 떠서 "원화 품질이 3D에 미치는 영향" 눈으로 확인.

## 3-3. 기준 설정값 (인수인계 확정치, 여기서 출발)
```
target_polycount: 3000     # 저폴리
should_remesh: true
should_texture: true       # 색 유지 (저폴리는 색이 생명, 특히 무늬종)
topology: quad
```
- 모델: **Meshy 5**(텍스처O=15크레딧) 먼저. 애매하면 Meshy 6(30크레딧) 비교.
- 밑면 중앙 피봇 옵션 있으면 켜기.

## 3-3★ 최종 확정값 (R1~R6 튜닝 완료, 2026-07-18 확정)
> 아래가 **대량변환 확정 설정**. 전 품목 통일, 무늬종도 예외 없음.
```
ai_model:         meshy-5        # 전 품목. Meshy 6 안 씀 (이유 아래)
target_polycount: 1500           # quad 기준 ≈ 3천 tri. T1·T2 다 형태 유지 검증됨
topology:         quad
should_texture:   true
should_remesh:    true
origin_at:        bottom
texture_prompt:   ❌ 안 씀        # 원본 무시하고 텍스트로 재생성 → 폐기 확정 (R3~R5)
texture_image_url: (무늬종만) 원본 이미지 공개URL 지정   # A안 방식. 텍스트 대신 이미지로 텍스처 유도
```
- **★ 잎 3D 각도 워크플로우 (정정 — 순서 중요)**:
  1. **입체(3/4) 원화로 먼저 3D 돌린다** ← 기본. 기존 원화 그대로 사용. 대부분 이걸로 잘 나옴(갤럭시 등 검증됨).
  2. 3D가 실패하면(컵휨/형태이상) → **같은 원화로 2회까지 재시도**.
  3. 그래도 안 되면 → **그때 flat 원화를 추가로 뽑아** 입체+flat **2장 multi-image**(`meshy_multi_image_to_3d`)로 재변환.
  - ❌ 처음부터 flat-first로 재생성하지 말 것. flat은 "3D 안 될 때 보강용 2번째 장"이지 1번 아님.

### ★ Meshy 6을 안 쓰는 이유 (핵심 교훈)
- 6은 무늬를 "또렷"하게 만드는데, 그 방식이 **색이 아니라 노멀맵(요철/단차)으로 형태에 새김.**
  → 줄무늬가 도톰하게 튀어나와 우글거림 = 우리가 PART 1-2에서 피하기로 한 **"무늬=평면" 규칙 위반.**
- **검증 근거**: 모든 meshy-6 산출물에 `_normal.png` 동반(요철 인코딩), meshy-5는 `base_color`만(평면 색).
- 5의 부드러운 물결 무늬 = 파스텔 저폴리(동물의숲st) 감성에 정확히 맞음. 채도도 가장 파스텔(sat 0.29).
- 5의 유일 약점 "무늬가 원본보다 부드럽게 풀림"은 게임 에셋으론 오히려 장점(사진st 또렷함 불필요).
- 뒷면 텍스처는 5가 6보다 약하나, **잎만 떼서 앞면 위주 배치**라 게임서 거의 안 보임 → 무관.

### R1~R6 검증 요약
- **요철 원화(T1)**: 저폴리 remesh가 평탄화함 → 원화 요철은 치명적 아님. 단 여전히 정면 평면 원화 권장.
- **컵휨(T2)**: 입체 원화로 먼저 시도가 기본. 컵휨 심하면 2회 재시도 → 안 되면 입체+flat multi-image (flat-first 아님).
- **폴리곤 1500**: 복잡한 몬스테라도 갈라짐·구멍·실루엣 유지. 확정.
- **texture_prompt**: image-to-3d에서 원본 이미지 텍스처링을 **대체**(공식문서 확인). muddy/색파탄/백지로 튐 → 폐기.
- **관찰된 한계** (대량변환 시 후처리 필요):
  - `origin_at:bottom`이 실제론 bbox **중앙** 정규화로 나옴(밑면 안착 X) → 배치 시 피벗 재정렬 필요.
  - 스케일이 최대변≈2.0 로 정규화됨 → 기존 `assets/monstera/` 에셋과 맞추려면 재스케일 필요.

## 3-4. 실험 순서
- **R1**: 기준값으로 T1·T2 각 1개 → GLB 받아 3-5 체크리스트로 채점. (핵심: T1 요철 남았나/평평해졌나)
- **R2**: 아쉬운 것만 변수 하나씩 — 폴리곤 1500/5000 비교 / Meshy 5 vs 6 / 텍스처 유무 / quad vs triangle
- **R3**: T1·T2 둘 다 만족하는 설정 1세트 확정 → 그 값으로 대량 변환
