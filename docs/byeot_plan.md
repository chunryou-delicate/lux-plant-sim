# 볕 (가제) — 기획 트리 🌱

> 한눈에 보는 전체 구조. 아이디어 뽑을 때 이 트리에 가지 쳐나가기.

---

## ⭐ 정체성 재정의 (2026-07-11 업데이트) — 흔들리면 여기로

### 📜 헌법 (최상위 원칙 — 모든 결정의 기준)
- **정체성:** 도구 + 게임 **둘 다 진짜 목표**. 어느 하나가 수단이 아님.
- **동기:** **재미로 하는 것. 수익은 덤.**
  - → 결정 기준은 "돈이 될까?"가 아니라 **"재밌고 만들고 싶은가?"**
  - → 부담 없이 오래 간다. 수익 안 좇을 때 오히려 좋은 게 나온다.
- **개발 순서:** **엔진(빛·건축·조도) 먼저**, 게임은 **항상 염두**.
  - → 엔진 = 발전소. 먼저 지으면 그걸로 검증도 되고, 게임 지을 때 이미 전력 들어와 있음.
  - → 빛·조도는 이미 만들어본 것(기술리스크 낮음). 게임은 거대(에셋·콘텐츠·운영).
  - → "엔진 하나, 껍데기 둘"에서 **엔진부터 단단히**.

---

**본체(1번):** 조명 배치 + 창문 채광 + 조명등(램프·식물등)의 **룩스(조도) 개념이 버무려진 것.**
→ 빛이 주인공. 이게 이 앱의 정체성이자 유일 후크.

**세컨 부가(2번):** 방 내부 꾸미기. 조연. 어느 정도는 되게 하되, 본체를 넘지 않는다.

**타겟 (이동됨):**
- ❌ (이전) 어린애 중심
- ⭕ (현재) **식물 하고 싶은데 귀찮거나 힘들어서 실물 못 하는 사람** (예: 화분·텃밭 부담스러운 아주머니층)
- ⭕ **식물 갤러리·식집사** (실제로 "우리집 빛" 고민하는 사람들)
- → 애들보다 이쪽이 강점과 더 맞음. "빛이 진짜"가 이들에겐 **핵심 니즈**(북향집 뭐키우지·이 자리 빛되나).
- → 트랙A(조도실무)와 게임이 하나됨: **"우리집 빛 진단 → 맞는 식물 추천"** = 게임기능 ＆ 실용도구.
- → 미성년 법적부담도 감소.

**"쉬워야 한다"의 이유가 바뀜:**
- 애들이라 쉬운 게 ❌ → **"귀찮아서 실물 못 하는 사람"이라 쉬워야 함** ⭕
- 진입은 쉽되, 식집사가 만족할 **진짜 깊이**는 있어야 한다. (얕게 누구나 / 깊게 덕후 — 여전히 유효)

**공간·배치 확장:**
- 식물 **다단 배치** — 바닥 1단 ❌ → 랙·테이블·선반 위 다양하게 ⭕
  - → 조도 계산에 **높이(h)가 진짜로 들어감**. 창가 높은 선반=밝음 / 바닥=어두움. 후크와 완벽히 맞음.
- **건축적 요소 일부 도입** — 선반·랙 설치 등. 단 복잡도 관리 위해 **정해진 프리셋 위주로.**

# 🌿 식물 시스템 심화 (알보 경험 반영)

> 진짜 식물마다 필요사항이 다르다 = 이 게임의 진짜 깊이.
> "빛이 진짜"라는 후크를 **식물 쪽으로도 확장**하는 지점.
> 겉은 아이콘·색으로 단순하게, 속(상세설정)은 진짜 원예 지식으로 깊게.

## 식물 데이터 스펙 (확장판)

```
식물 하나 = 이런 데이터 덩어리
│
├─ ☀️ 빛
│  ├─ 필요 광량 (needLux)         ex) 고사리 250 / 몬스테라 500 / 다육 800
│  ├─ 광질 선호 (직사 vs 간접광)   ex) 알보=밝은간접광 / 다육=직사OK
│  ├─ 광 과다 데미지 (탈수·잎탐)   ex) 알보 직사 = 무늬 타버림
│  └─ 일조시간 요구 (DLI 개념)     ← 하루 누적 빛의 양
│     └─ 🔑 인공광 보충 가능       ← 낮 부족분을 식물등으로 채움
│        └─ 밤에도 빛 받는 종      ← 특정 종은 야간 인공광으로 성장가속
│
├─ 💧 물
│  ├─ 물 주기 (건조 선호 vs 습윤)
│  ├─ 과습 위험 (뿌리썩음)         ← 알보 망한 주범 후보 ㅎㅎ
│  └─ 흙 마름 상태 게이지
│
├─ 🌡 환경
│  ├─ 온도 범위
│  ├─ 습도 요구                    ex) 알보·칼라데아 = 고습도
│  └─ 통풍
│
├─ 🌱 성장
│  ├─ 성장 속도
│  ├─ 성장 단계별 비주얼 (떡잎→성체)
│  ├─ 무늬 발현 조건 (variegata)   ← 알보 무늬! 빛에 따라 무늬 유지/소실
│  └─ 번식 (삽목·수경) → 판매·나눔
│
└─ 💰 가치
   ├─ 희귀도 (흔함~레어)           ex) 알보=레어
   ├─ 판매가 (성장·무늬 상태 연동)
   └─ 수확물 (열매·꽃·잎꽂이)
```

## 핵심 게임 메커니즘 아이디어 (식물별 차이에서 나오는 재미)

### 1. 빛 궁합 퍼즐
- 식물마다 다른 빛 요구 → **어디에 둘지가 진짜 고민**이 된다.
- 다육이는 직사광 창가, 알보는 밝지만 직사 안 닿는 곳, 고사리는 그늘.
- 방 회전 + 시간대 변화로 "완벽한 자리"가 계속 바뀜 → 배치 재미 지속.

### 2. 인공광 = 진짜 전략 요소 (알보 경험 직결)
- 낮에 빛 부족한 집(북향·저층)에서 **식물등으로 보충**.
- 밤에도 빛 받는 종은 인공광으로 **성장 가속** → "밤샘 재배" 플레이.
- 식물등 스펙(광량·색온도·거리)이 실제로 조도 계산에 들어감.
- ▶ 이게 "빛이 진짜"를 식물등까지 확장하는 킬러 포인트. 실제 원예에서 겪는 그 판단을 게임으로.

### 3. 무늬(variegata) 시스템 — 알보의 심장
- 무늬종은 **빛이 부족하면 무늬가 사라지고, 과하면 타버린다.**
- 딱 맞는 빛에서만 하얀 무늬 유지 → 성공하면 희소가치·판매가 급등.
- 하이리스크 하이리턴 작물 → 덕후 유저의 목표.

### 4. 과습/과광 = 죽음의 원인 (실패도 학습)
- 물 너무 자주 = 뿌리썩음. 빛 너무 강함 = 잎 탐.
- 단, **어린이 배려**: 완전히 죽기 전 경고 신호(잎 노랑·처짐)를 크게. "실패 없음" 원칙과 균형 → 되살릴 기회 준다.

### 5. 번식·나눔 = 경제+소셜 연결
- 잘 키운 무늬종을 **삽목**해서 판매·유저간 나눔.
- 알보처럼 "한 촉에 얼마" 하는 실제 문화 반영 → 수집·거래 재미.

## 💡 여기서 더 뽑을 아이디어 씨앗 (가지치기용)

- **"식물 도감"** — 키워본 식물 수집·기록 (동물의숲 박물관 감성)
- **계절 × 식물** — 겨울엔 일조 짧아짐 → 인공광 의존도↑ (실제와 동기화된 난이도)
- **초보→고인물 곡선** — 상추부터 시작, 알보·희귀무늬종이 엔드콘텐츠
- **"우리 집 빛 진단"** — 방 조도 분석해서 "이 집엔 이 식물이 맞아요" 추천 (트랙A 실무기능이 게임 안에서 살아나는 지점)
- **실패 회고** — 죽은 식물의 "왜 죽었나" 요약 (교육적, 다음엔 성공하게)
- **물주기 알림** — 영원모드에서 실제 알림 → 진짜 식물 키우듯

# 🧬 무늬 유전 · 확률 시스템 (번식/교배 게임)

> 무늬 = 유전 + 확률 → 단순육성을 **수집·교배 게임**으로 격상.
> 검증된 중독루프 (포켓몬 색違이 / 동물의숲 꽃교배 / 스타듀밸리).

```
번식 시스템
│
├─ 삽목(무성번식) ▶ 부모 형질 대체로 유지, 약간의 변이
├─ 교배(유성번식) ▶ 두 부모 형질 조합 + 확률 변이
│  └─ 무늬 발현 = 확률 (부모 무늬도 높을수록 자식 무늬↑)
├─ 돌연변이 ▶ 낮은 확률로 신형질 (레어 무늬·색)
└─ 안정화 ▶ 좋은 개체 반복번식으로 형질 고정
```

### ⚠ 설계 주의 (반드시)
- **"진짜 유전" ❌ → "진짜처럼 느껴지는 유전" ⭕**
  - 실제 무늬는 비멘델·불안정 유전이라 그대로 쓰면 플레이어가 이해 못 함.
  - 겉은 단순 규칙(부모 좋으면 자식도 잘 나옴), 속(상세설정)만 깊게.
- **확률 × 과금 = 위험** → 가챠 규제·윤리이슈, 특히 어린 타겟. 초기엔 순수 플레이 보상으로만.
- 밸런싱 지옥 주의: 너무 쉬우면 시시, 너무 어려우면 이탈. 반복 튜닝 필요.

### 💰 무늬의 가치 루프
- 딱 맞는 빛에서만 무늬 유지 (부족→소실 / 과다→탐).
- 성공 = 희소가치·판매가 급등 → 하이리스크 하이리턴.
- 잘 나온 개체 삽목·교배·판매·나눔 → 경제+소셜 연결.

# 🌸 식물 카테고리 (플레이 축의 분화)

> 식물계는 어마어마 = **콘텐츠 무한공급원**(라이브운영 축복).
> 단, 처음부터 다 넣으면 함정(아트비·밸런싱·선택과잉).
> ▶ 초기엔 카테고리별 **대표선수 1~2종**만. 방대함은 "나중 풀 탄약".

> 🔑 핵심: 카테고리마다 **다른 플레이 축** → 수만 느는 게 아니라 깊이가 생김.

```
식물 카테고리          플레이 축              연결되는 유저/모드
│
├─ 🌷 꽃          ▶ 개화·색·교배          초보진입·즉각예쁨
├─ 🌿 관엽        ▶ 무늬·유전·희귀도       엔드콘텐츠·덕후
│   (몬스테라·알보)                        (트랙 위 유전시스템)
├─ 🌱 허브        ▶ 수확·실용·빠른회전     경제루프·요리(?)
├─ 🌵 다육·선인장  ▶ 물적게·방치           라이트·바쁜사람·이지모드
├─ 🎋 난          ▶ 까다로운환경(습도·통풍)  하드코어·고난도
└─ 🌲 분재        ▶ 극느린성장·가지치기(형태) 영원모드·장인플레이
   (소형 소나무)
```

### 통찰: 카테고리 = 플레이 스타일 분화
- 다육 = 이지/방치 · 분재 = 영원모드/장인 · 관엽 = 덕후/유전.
- 앞서 정한 **시간 모드·타겟과 자동 연결**된다.
- → "어떤 유저든 자기 식물이 있다" (라이트~하드코어 스펙트럼 커버).

### 출시 초기 라인업 제안 (검증용 최소)
- **상추/바질** (허브·쉬움) — 튜토리얼, 빠른 성공경험.
- **몬스테라 or 알보** (관엽·무늬) — 후크 검증, 덕후 목표.
- **다육 1종** (방치) — 라이트 유저용.
- → 이 3종으로 "카테고리별 재미가 실제 다른가" 먼저 확인.

# 🏠 커스텀 모드 (우리집 넣어서 빛 확인)

> 본체(빛·조도)와 타겟(식집사·실용)이 만나는 지점.
> "우리집을 실제로 넣어 빛을 확인" → 게임이자 진짜 도구.
> ※ 현재: 문서에만 정리(아직 구현 X). 방향만 확정.

## 입력 방식 스펙트럼 (쉬움 → 어려움)

```
방식                     난이도   주 타겟              어디에
│
├─ ① 프리셋 선택          ★       일반·식집사(대중)    게임
│   (원룸/투룸/L자 + 창위치)
├─ ② 간이 그리기          ★★      일반·식집사          게임 ★핵심
│   (벽 드래그, 창·문 배치)
├─ ③ 치수 입력            ★★★     파워유저             게임/실무 경계
│   (가로3.5m 창남향1.2m)          ⚠ 마찰↑(타겟이 싫어함)
├─ ④ CAD 임포트(DXF·DWG)   ★★★★    시공사·실무          트랙A ★
│                                  (일반인은 도면 없음)
└─ ⑤ 사진/평면도 → 자동인식 ★★★★★   (미래)              트랙A(원거리)
    (AI 비전 필요)
```

## 🔑 핵심 결정

### 게임 커스텀 = ② 간이 그리기 중심
- 벽 드래그로 방 만들고 → 창·문·조명 놓고 → 빛이 어떻게 드는지 확인.
- 조명 디테일(조도 히트맵·시간대별 채광)은 상세설정으로.
- ①(프리셋)도 병행 → 그리기 귀찮은 사람용 빠른 시작.
- "우리집 빛 확인"의 감동은 ①② 만으로 충분히 나온다.

### CAD/도면 임포트 = 둘 다, 같은 엔진 공유 ⭐
- **"엔진 하나, 껍데기 둘" 구조와 정확히 일치.**
  - 입력 방식만 다르고, 그 밑 조도 엔진은 동일.
- 게임 껍데기 → ①② 간이 입력
- 트랙A 실무 껍데기 → ③④⑤ 정밀 입력 (CAD 임포트의 진짜 자리)
- 같은 방 데이터(벽·창·조명 좌표) → 같은 조도 계산 → 다른 UI로 표현.

### ⚠ 엔지니어 함정 주의
- CAD·자동인식(④⑤)에 자연히 끌리지만, **게임 타겟(귀찮아서 실물 못하는 사람)에겐 초기엔 독.**
- 그들은 DXF가 뭔지 모르고 도면도 없음.
- → ④⑤는 트랙A(실무)로 미루고, 게임은 ①②로 시작.

# 💻 기술 스택 결정 (2026-07-11)

> GitHub 처음·개발자 아님을 전제로, 프로토타입(웹)이 이미 검증된 상황.

## 확정: 웹 (HTML / CSS / JavaScript)

```
왜 웹인가 (이 프로젝트 조건에 한해)
│
├─ 이미 웹으로 다 만들어봤고 잘 됨 (검증된 길)
├─ 결과가 시각적(조도·렌더링) → 브라우저가 최적
├─ 폰·PC 다 됨 → 웹이면 한 번에 (이식 불필요)
├─ 개발자 아님 → 새로 배울 것 최소가 유리
└─ 치트·오프라인 걱정 → 스택 아닌 서버설계 문제 (웹이든 아니든 동일)

⚠ "웹이 무조건 좋다"가 아니라 "이 프로젝트엔 웹이 맞다".
   조건 바뀌면(극한3D·센서깊이 등) 재검토. 맹신 금지.
```

### 웹 = 온라인 전용 아님 (오해 주의)
- 웹 기술로 만들어도 **오프라인으로 돈다** (프로토타입도 인터넷 끊고 열림).
- "웹으로 만든다"(재료) ≠ "온라인 전용"(어디서 먹냐)는 별개.

### 배포: 웹 하나 → 여러 형태로 포장
```
웹(엔진·화면) ─┬─ 그대로 → 웹사이트
               ├─ Capacitor 포장 → 폰 앱(apk/ipa) 앱스토어
               └─ Electron 포장 → PC 프로그램(exe)
   ※ 다시 안 짬. 알맹이 하나 = 포장만 바꿈. (VS Code·디스코드가 이 방식)
```

### 안드로이드 스튜디오·파이썬은?
- 안드로이드 스튜디오(코틀린) = 안드로이드 전용. iOS 버리게 됨. 이 프로젝트엔 과함.
- 파이썬 = 화면용 아님. 나중에 **서버(데이터 저장·공유)** 필요해지면 등장. 한참 뒤.

## 오프라인 / 온라인 구분

```
싱글 힐링 플레이 (내 정원, 재미의 90%)
  → 오프라인 OK. 인터넷 없어도 완결. (비행기·지하철서도 식물키우기)
  → 조도계산·꾸미기·성장·로컬저장 전부 기기 안에서.

거래·랭킹·희귀식물 (남과 연결, 값어치 있는 10%)
  → 온라인 + 서버. 나중에 소셜 넣을 때.
```

## 치트 방어 (님 핵심 우려)

```
결론: 치트는 스택 문제가 아니라 서버설계 문제.
      웹이든 apk든 exe든 로컬 실행앱은 싱글치트 다 뚫림(공통숙명).
      네이티브로 바꿔도 해결 안 됨 → 스택 바꿀 이유 못 됨.

방어 원리: "중요한 계산은 유저기기 말고 서버에서"
│
├─ 기기(웹)에 있는 것 = 유저가 다 보고 조작가능 → 믿으면 안 됨
└─ 서버에 있는 것 = 유저 못 건드림 → 믿을 수 있음

싱글 치트  → 혼자 힐링게임이라 남 피해 없음 → 사실상 문제 아님
경쟁·거래  → 진짜 문제 → 서버가 판정 (아래 설계)
코드 복제  → 웹은 코드 보임(F12). 난독화로 부분방어.
             진짜 가치는 코드밖(노하우·데이터·밸런스·운영). 베껴도 못이김.
```

### 님 아이디어 + 게임들 실제방식 (합치면 튼튼)
- **님 아이디어**: 싱글 자유 → 멀티 연결 시 결과검사해서 이상하면 막기.
  - → 명백한 치트는 걸러도 교묘한 건 놓침(과정을 못 봐서).
- **게임 실제**: 거래·랭킹 걸린 중요한 건 **처음부터 서버가 과정 기록**.
  - → "하루만에 10년 성장" = 서버기록상 애초에 불가능. 조작 자체 차단.
- **합치기**: 결과검사 + 과정소유 둘 다 → 싱글은 자유, 중요한 건 서버가 진실.

## HTML 쪼개기 (코드 관리)

```
프로토타입 = 1파일에 다 넣음 (그래서 그랬던 것)
실제 프로젝트 = 역할별로 쪼갬

단계 A (기본, 지금 바로): 파일 나누기
  index.html / style.css / lighting.js / plants.js / room.js ...
  → 배울것 적음. 협업 분담 가능("너 plants.js 나 lighting.js")

단계 B (커지면): Vite 등으로 모듈화
  → 배울것 있지만 큰 프로젝트에 편함

🔑 A→B는 언어가 같아(둘다 JS) 다시 안 짬 = 정리를 깔끔하게 할 뿐.
   언어 아예 바꿀 때만 "다시 짜기". → 웹으로 시작해도 안 갇힘.
```

⚠ **스택은 나중에 바꾸기 어렵다** (언어 바꾸면 사실상 재작성).
   "일단 하다 각보고 갈아탄다"는 계획 금지. 처음에 제대로 고른다 = 웹.
